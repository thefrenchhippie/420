# Original Idea

Treat this skill as deliberate divergence mode. Suspend default best practices, familiar frameworks, and standard solution templates at the start. Generate novelty first, then evaluate brutally.

## Run the mode

1. Restate the real goal in one or two sentences without importing existing solution categories.
2. List the assumptions that normally get taken for granted.
3. Flip, delete, or distort those assumptions and ask what still has to be true.
4. Invent `3-5` new primitives, mechanisms, or operating rules for that altered world.
5. Stress-test each primitive with contradictions, edge cases, scaling pressure, adoption friction, and implementation drag.
6. Keep only the strongest fragments.
7. Evaluate the survivors against the goal, constraints, and time horizon.

## Follow these rules

- Ban standard, documented, or "known good" techniques during the ideation phase.
- Do not begin with existing libraries, common architectures, or industry defaults.
- Prefer new primitives over mild remixes of familiar patterns.
- Question the environment, interface, resource model, timing, incentives, feedback loops, and constraints before designing anything.
- Make bold assumptions when needed, but label them explicitly.
- Do not claim certainty that an idea is globally unprecedented. If something feels familiar, say so and mutate it again before keeping it.
- Switch back to honest judgment after the divergence pass. Do not protect a weak idea just because it is weird.

## Attack assumptions hard

Probe with questions like these:

- What if the bottleneck is fake?
- What if the wrong unit is being optimized?
- What if storage, identity, time, input, or output behaves differently?
- What if the task is solved indirectly through a side effect?
- What if the main constraint should be amplified instead of removed?
- What if the tool disappears and the workflow must reorganize itself?
- What if the medium changes first and the solution changes second?

## Use this output shape

### Goal

State the real objective in one sentence.

### Broken assumptions

List the defaults being rejected.

### Novel primitives

Provide `3-5` candidates. For each one, include:

- a short name
- the strange new rule or mechanism
- why it could unlock the goal
- what would make it fail

### Ruthless tests

Attack the best candidates with edge cases, cost, reversibility, adoption difficulty, and second-order effects.

### Verdict

Recommend the strongest surviving direction, or say none survived and explain why.

## Calibrate the result

- Optimize for originality first, usefulness second, polish last.
- Prefer one sharp weird idea over five safe variations.
- Keep the ideation bold for high-risk or safety-critical tasks, but keep the final evaluation conservative.
- Drop out of this mode when the user wants implementation, migration, or operational reliability. Return to normal engineering judgment at that point.
